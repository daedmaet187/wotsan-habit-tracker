## Secrets Management

### How Secrets Are Generated

Secrets are generated by Terraform's `random` provider — **cryptographically secure** and reproducible via Terraform state:

| Secret | Resource | Spec |
|--------|----------|------|
| DB password | `random_password.db_password` | 32 chars, upper+lower+numeric+special, excludes `@/\` (breaks connection strings) |
| JWT secret | `random_bytes.jwt_secret` | 32 bytes → 64-char hex (256-bit entropy) |

No human ever chooses these values. They are generated on first `tofu apply` and stored in state + Secrets Manager.

### Where Secrets Are Stored

**AWS Secrets Manager** (encrypted at rest with KMS default key, AES-256):

| Secret Name | Secrets Manager Path |
|-------------|---------------------|
| DB password | `wotsan-habit-tracker/db-password` |
| JWT signing secret | `wotsan-habit-tracker/jwt-secret` |

The plaintext values are **never visible in the AWS ECS console** — unlike plain `environment` variables.

### How ECS Accesses Secrets

ECS Fargate pulls secrets at container start via the `secrets` field in the task definition:

```json
"secrets": [
  { "name": "JWT_SECRET", "valueFrom": "<secrets-manager-arn>" }
]
```

The ECS execution role has `secretsmanager:GetSecretValue` + `secretsmanager:DescribeSecret` scoped to only the two ARNs needed. The container never makes a direct API call — ECS injects the value as an environment variable before the process starts.

`DATABASE_URL` remains in `environment` (not `secrets`) because it's a constructed string that already embeds the password. Moving it to Secrets Manager would require the app to parse it differently — the security benefit is marginal since it's already protected by the state backend's S3 encryption.

### Secret Rotation

**Manual rotation procedure:**

1. **JWT Secret** (no downtime required):
   ```bash
   cd infra
   tofu state rm random_bytes.jwt_secret
   tofu apply  # regenerates new secret, updates Secrets Manager, triggers ECS redeploy
   ```
   All existing JWT tokens become **immediately invalid** — users must re-login.

2. **DB Password** (brief downtime):
   ```bash
   cd infra
   tofu state rm random_password.db_password
   tofu apply  # regenerates password, updates Secrets Manager, updates RDS, redeploys ECS
   ```
   RDS password change + ECS redeploy happen in the same apply. Monitor for a brief connection interruption.

### What's Still in GitHub Secrets

These cannot be eliminated without OIDC:

| Secret | Purpose | Risk |
|--------|---------|------|
| `AWS_ACCESS_KEY_ID` | Terraform authentication | Long-lived credential — see OIDC upgrade path |
| `AWS_SECRET_ACCESS_KEY` | Terraform authentication | Long-lived credential — see OIDC upgrade path |
| `CF_DNS_TOKEN` | Cloudflare DNS management | Scoped to DNS edit only |
| `CF_ZONE_ID` | Cloudflare zone identifier | Not sensitive, but kept consistent |

**Removed from GitHub Secrets** (no longer needed):
- `DB_PASSWORD` — now generated by Terraform
- `JWT_SECRET` — now generated by Terraform

### OIDC Upgrade Path (Future)

Replace `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` with GitHub OIDC federation:

```yaml
permissions:
  id-token: write
  contents: read

- uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::ACCOUNT:role/github-actions-opentofu
    aws-region: eu-central-1
```

This eliminates long-lived AWS credentials from GitHub Secrets entirely. Effort: Medium. Priority: High for production.

## RDS Security

| Property | Value | Notes |
|----------|-------|-------|
| Public access | `false` | Only accessible from within VPC |
| Subnet placement | Private subnets only | `db_subnet_group` uses private subnet IDs |
| Security group | Port 5432 from ECS SG only | No CIDR-based ingress from public internet |
| Storage encryption | `storage_encrypted = true` | AES-256, KMS default key |
| Deletion protection | `deletion_protection = true` | Prevents accidental `tofu destroy` |
| Backup retention | 7 days | Daily automated snapshots |
| Backup window | `03:00-04:00 UTC` | Low-traffic window |
| Maintenance window | `Mon 04:00-05:00 UTC` | Immediately after backup window |
| Password | 32 chars, special chars | Never visible in AWS console or ECS environment |
| Final snapshot | `skip_final_snapshot = true` | Dev/staging acceptable; change to `false` for prod |

> ⚠️ `skip_final_snapshot = true` is intentional for cost — this is a personal project. For any production data you care about, set to `false`.

## ECS / API Security

### Container Environment

Sensitive values use the ECS `secrets` field (fetched from Secrets Manager at start, never shown in console):

| Variable | Method | Reason |
|----------|--------|--------|
| `JWT_SECRET` | `secrets` field | Signing credential — must never appear in console |
| `DATABASE_URL` | `environment` field | Constructed string; state-level encryption covers it |
| `PORT`, `NODE_ENV`, `JWT_EXPIRES_IN` | `environment` field | Non-sensitive config |

### CloudWatch Logs

- Log group: `/ecs/wotsan-habit-tracker-api`
- Retention: **7 days** (balance between cost and audit trail)
- All `console.log` output goes here — ensure no passwords/tokens are logged in app code

### Networking Gap

> **Known gap:** `assign_public_ip = true` with tasks in public subnets

Tasks have direct internet exposure via their public IP. This is intentional for cost (NAT gateway = ~$32/month). The ALB sits in front and the ECS security group only allows inbound on port 3000 from the ALB security group, mitigating most risk. See [Known Security Gaps](#known-security-gaps).

### Image Scanning

ECR image vulnerability scanning on push is **not yet configured** (see Known Gaps). Add via:

```hcl
resource "aws_ecr_repository" "main" {
  image_scanning_configuration {
    scan_on_push = true
  }
}
```

## API Layer Security

The Node.js API applies defense-in-depth:

| Layer | Implementation | Config |
|-------|---------------|--------|
| Security headers | `helmet.js` | XSS protection, HSTS, CSP, X-Frame-Options |
| CORS | Restricted origin list | `habit-admin.stuff187.com`, `localhost:5173` only |
| Rate limiting (global) | `express-rate-limit` | 200 req / 15 min per IP |
| Rate limiting (auth) | `express-rate-limit` | 10 req / 15 min per IP on `/auth/*` |
| Input validation | `zod` | All mutation endpoints validated before DB touch |
| Authentication | JWT HS256 | Secret from Secrets Manager, 7d expiry |
| Password hashing | `bcrypt` | Cost factor 12 (~250ms hash — brute-force resistant) |
| Error handling | `errorHandler.js` | Raw errors suppressed in production responses |
| SQL injection | Parameterized queries | Never interpolate user input into SQL strings |

## JWT Best Practices

### Algorithm

`HS256` (symmetric HMAC-SHA256). Appropriate for this architecture — single service, no distributed token verification needed. Use RS256 only if multiple services need to verify tokens independently.

### Expiry

`7d` — balances UX (less re-login friction) vs security (shorter window for stolen token abuse). If implementing refresh tokens in future, reduce access token expiry to `15m`.

### Secret Entropy

64-char hex (256-bit entropy) generated by `random_bytes`. Computationally infeasible to brute-force.

### Token Storage

| Client | Storage | Risk Level | Acceptable? |
|--------|---------|-----------|-------------|
| Mobile (Flutter) | `SharedPreferences` | Low | ✅ Yes — sandboxed per app |
| Admin web | `localStorage` | Medium | ✅ Acceptable for internal admin tool |

`localStorage` is vulnerable to XSS. Helmet.js CSP headers mitigate this. For public-facing apps, prefer `httpOnly` cookies.

### No Refresh Tokens (Current)

Current implementation uses single long-lived access tokens. To improve:
1. Issue short-lived access tokens (15m)
2. Issue long-lived refresh tokens (30d) stored securely
3. Implement `/auth/refresh` endpoint

This reduces the blast radius of a stolen access token.

### Rotation Procedure

If JWT secret is compromised:
1. Update the Secrets Manager value (or taint + tofu apply to regenerate)
2. Force ECS task redeploy to pick up new secret
3. **All existing tokens become immediately invalid** — all users must re-login
4. Notify users if appropriate

## AI Agent Security Checklist

> **Codex / AI agents: read this section before implementing any feature.**

### New API Endpoint

- [ ] Zod schema validates all body fields
- [ ] Auth middleware applied (`auth.js`)
- [ ] Admin-only routes have `adminAuth` middleware
- [ ] No raw error messages in responses (`errorHandler.js` handles this)
- [ ] Rate limiting covers the route (global limiter applies unless explicitly exempt)
- [ ] No user-supplied data interpolated directly into SQL (use parameterized queries only)
- [ ] No sensitive data logged (no passwords, tokens, or PII in `console.log`)

### New Secret / Credential

- [ ] NEVER hardcode secrets in source code
- [ ] NEVER commit `.env` files (`.gitignore` covers `.env` — verify before committing)
- [ ] New secrets go in AWS Secrets Manager (add `aws_secretsmanager_secret` resource in `infra/main.tf`)
- [ ] ECS accesses via `secrets` field in task definition, not `environment`
- [ ] Local dev only: `.env` file (gitignored) — document required vars in `.env.example`
- [ ] Document in `playbook/SECURITY.md`

### New Infra Resource

- [ ] Sensitive variables marked `sensitive = true` in `variables.tf`
- [ ] No sensitive output values logged in CI (use `sensitive = true` on outputs)
- [ ] S3 buckets: block all public access unless explicitly a public asset bucket
- [ ] RDS: `publicly_accessible = false` always
- [ ] Security groups: minimum required ports only, source-restricted (not `0.0.0.0/0` for ingress)

### Mobile App

- [ ] No secrets in source code (use `dart-define` for build-time config)
- [ ] `API_URL` as `dart-define`, not hardcoded string
- [ ] Token stored in `SharedPreferences` (not in memory only — survives restart)
- [ ] Token cleared on logout (`auth_provider.dart` handles this)
- [ ] HTTPS only (no `http://` API URLs in production)

### Admin Frontend

- [ ] `VITE_API_URL` from env var, not hardcoded
- [ ] No API keys or secrets in frontend source
- [ ] CSP headers served by CloudFront (TODO — not yet implemented)
- [ ] Auth token in `localStorage` — acceptable for admin-only internal tool

## Known Security Gaps

Honest inventory of outstanding issues. Fix priority: High > Medium > Low.

| Gap | Risk | Effort | Recommended Fix |
|-----|------|--------|-----------------|
| ECS tasks in public subnets (`assign_public_ip = true`) | Medium | High | Move tasks to private subnets + NAT gateway (~$32/month). ALB stays public. Current mitigation: ECS SG only allows inbound from ALB SG. | 
| No ECR image vulnerability scanning | Medium | Low | Add `scan_on_push = true` to ECR repository resource in `modules/ecr/main.tf`. Free with ECR Basic Scanning. |
| No AWS WAF on ALB | Medium | Medium | Add `aws_wafv2_web_acl` + `aws_wafv2_web_acl_association`. Protects against SQLi, XSS, bad bots at the edge. Cost: ~$5-15/month. |
| No refresh tokens / short-lived access tokens | Low | Medium | Implement refresh token rotation. Reduces blast radius of stolen tokens from 7d to 15m. |
| No CloudTrail logging | Low | Low | Enable CloudTrail in the AWS account (not Terraform-managed usually). Provides API call audit trail for incident response. |
| GitHub Actions uses long-lived IAM keys | High | Medium | Migrate to GitHub OIDC federation. Eliminates `AWS_ACCESS_KEY_ID`/`SECRET` from GitHub Secrets entirely. See [OIDC Upgrade Path](#oidc-upgrade-path-future). |
| No CSP headers on admin CloudFront distribution | Low | Low | Add CloudFront response headers policy with strict CSP. Reduces XSS risk for admin localStorage token. |
| No RDS automated password rotation Lambda | Low | Medium | AWS Secrets Manager supports automatic rotation via Lambda. Currently rotation is manual (`tofu state rm` + apply). Implement if compliance requires regular rotation. |
| `skip_final_snapshot = true` on RDS | Medium | Low | Set `skip_final_snapshot = false` and define `final_snapshot_identifier` if data persistence matters. Currently acceptable for personal project. |

### Risk Definitions

- **High**: Directly exploitable, data breach possible, or compliance violation
- **Medium**: Increases attack surface or blast radius significantly
- **Low**: Defense-in-depth improvement; not directly exploitable with current setup
